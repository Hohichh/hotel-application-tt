# Уровень доступа к данным (Repository layer)

Здесь определяются интерфейсы JPA репозиториев (extends JpaRepository<Entity, IdType>) а также методы для работы с БД.

Использование интерфейсов позволяет гибко подменять реализации спецификаций орм и драйверов к бд, что опять же дает возможность легко "переключаться" между бд - достаточно поменять используемый профиль, где будет ссылка на датасурс

## Методы и типы queries

1. GET /hotels - здесь, лучше сделать не просто naming query, а сделать проекцию. Добавим interface - например HotelShort который а также на SpEL напишем маппинг, чтобы собрать поля следующим образом: "9 Pobediteley Avenue, Minsk, 220004, Belarus", то есть номер дома, улица, город, индекс, страна.

2. GET /hotels/{id} - извлечь полную entity Hotel, а также загрузить amitites. Для корректной загрузки amities можно использовать аннотацию @EntityGraph, указав в атрибуте attributeName "amenities". В качестве альтернативы можно использовать JPQL запрос с fetch join. !!Надо полумать что лучше!!
Еще тут стоит использовать optional. В Service будем проверять на null и кидать 404 - поскольку если id есть а ресурса нет - это скорее критичное исключение.

3.  GET /search?name=...&brand=...&city=...&country=...&amenities=...
- Тут у нас просто фильтрация по одному или более параметрам. Например, все отели Hilton в Palm Beach. Тут пригодится Specification API, я думаю. Хотя может будет лучше руками написать запрос с OR? По времени/памяти вероятно оптимальнее.
- еще здесь тоже краткие проекции, как в п.1, тоже смаппим.

4. POST /hotels
Тут надо смотреть, умеет ли hibernate сам резолвить many-to-many. Но скоре всего да - то в сервисе будем маппить dto в entity и сохранять. Короче надо убедиться, что он сможет добавить пару ключей, связывающих отель и amenities. 
Если amentities не существует, то надо бы кинуть expetion. Я бы сделала отдельный энд-поинт для их добавления, чтобы избежать случайного создания отеля с несуществующими amenities. Потенциально это важный buisness-rule

5. POST /hotels/{id}/amenities - добавление списка amenities к отелю
Тут по сути похоже на п.4, добавить ключи в таблицу hotel_amenities. По скольку у нас unique констрейнт, то без перезаписей. В будущем добавить проверку (в сервис) что отель существует. 

6. GET /histogram/{param} - здесь предполагается 1 конкретный параметр группровки, поскольку испольузется не requestparam, а pathvariable. На уровне маршрутов разрулим потом. Вроде мы можно сделать параметризованный запрос с group by. Правда тут надо глянуть, как обезопасить от sql-инъекций, если параметр будет передан извне. Ну либо провреять выше, на соотвествие параметра заданному словарю к примеру - и откидывать все запросы, которые содержат что-то кроме city, brand, country, amenities.
